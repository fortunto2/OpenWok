## Failure Patterns

### Pattern 1: Deploy Spin-Loop (AskUserQuestion in Pipeline)
- **Occurrences:** 5 iterations (iter 4-8)
- **Root cause:** Deploy skill encountered no deploy CLI tools, no Dockerfile, no fly.toml. Instead of making autonomous decisions in pipeline mode, it asked the user via AskUserQuestion in every iteration. Responses don't persist across pipeline iterations, so each iter started from scratch.
- **Wasted:** 5 iterations (56% of total pipeline)
- **Progression:**
  - Iter 4-6: Check -> find nothing -> AskUserQuestion ("Куда деплоим?") -> exit (same pattern 3x)
  - Iter 7: Finally autonomous — created Dockerfile/fly.toml/.dockerignore, pushed 18 commits, but couldn't auth to Fly.io
  - Iter 8: Installed flyctl, still couldn't auth -> AskUserQuestion
  - Iter 9: Attempted fly auth (opened browser), wrote state file to exit loop
- **Circuit breaker gap:** Iterations 4-6 asked the same question with slightly different wording each time, so the fingerprint-based circuit breaker (md5 of last 5 lines) didn't trigger. The content varied enough to evade identical-failure detection.
- **Fix:** Deploy skill needs `--dangerously-skip-permissions` pipeline mode detection. When no CLI/auth available: create configs, push code, mark "ready for manual deploy" via state file — never AskUserQuestion.

### Pattern 2: Spec Criteria Not Updated
- **Occurrences:** 1 (post-build)
- **Root cause:** `/build` updated `plan.md` task checkboxes but did not update `spec.md` acceptance criteria. All 10 criteria in spec.md remain `- [ ]` despite all work being complete.
- **Wasted:** 0 iterations (cosmetic, but reduces fidelity tracking accuracy)
- **Fix:** `/build` skill should mark spec.md criteria as `[x]` when corresponding tasks complete. Already flagged in supervox retro and life2film runs 3-6.
